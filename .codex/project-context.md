# 项目背景

## 1. 项目身份

- 当前仓库已经不是原始的 Web 优先项目 `Kite`。
- 当前产品目标是桌面应用：`Kite Desktop`。
- 桌面端基于 `Wails v3` 构建，并在本机 `127.0.0.1` 启动一个内嵌后端服务。
- 后续开发时，不要默认把它当成一个纯浏览器 / 纯服务端项目来设计。
- 仓库中仍保留了一些服务端 / Web 入口，但它们已经不是当前产品的第一优先级。

## 2. 核心架构

当前应用由四部分组成：

1. Go 后端服务
   - 负责 Kubernetes 访问、API 处理、持久化、AI 集成等核心逻辑。
   - 主要代码在 `internal/`、`pkg/`、`cmd/`。

2. 共享 React 业务前端
   - 主业务 UI 位于 `ui/`。
   - 这是桌面应用里真正展示给用户的主要界面。
   - 当前使用 React 19。

3. Wails 桌面宿主
   - 桌面入口位于 `desktop/`。
   - 负责启动本地后端、管理窗口、托盘、原生文件对话框，以及各类桌面专属能力。

4. 桌面壳层前端
   - `desktop/frontend/` 是给 Wails 壳层使用的小型前端，不是主业务 UI。
   - 当前使用 React 18。
   - 只有在桌面壳层 / 启动壳 / 引导页相关需求下，才应该改这里。

## 3. 关键入口文件

- `main.go`
  - 传统服务端入口。
- `desktop/main.go`
  - 真正的桌面应用入口。
- `internal/server/app.go`
  - 负责初始化后端运行时并构建 Gin 引擎。

## 4. 运行时模型

桌面运行时是显式定义的，不是隐式推断的。

- 桌面运行时标识：`APP_RUNTIME=desktop-local`
- 本地后端只监听 loopback 地址
- 桌面宿主会在后端启动前注入桌面运行所需环境变量
- 桌面模式默认被视为单机、本地、单用户工具

涉及运行时差异时，优先阅读：

- `docs/desktop-runtime-contract.md`
- `docs/desktop-feature-boundary.md`
- `desktop/README.md`

## 5. 必须坚持的产品前提

- 当前项目应优先采用桌面原生流程，而不是浏览器专属流程。
- 桌面与 Web 的行为差异，应尽量通过运行时能力判断、后端契约或桌面宿主层来实现。
- 不要在页面组件里到处分散写浏览器假设或零散环境判断。
- 如果某个功能涉及桌面原生能力，应优先检查是否应该接入：
  - `ui/src/lib/desktop.ts`
  - `desktop/bridge.go`
  - `desktop/` 下的宿主层代码

典型例子：

- 外链通过桌面 bridge 打开
- 文件保存 / 下载走原生对话框
- 打开配置目录 / 日志目录走宿主能力
- 关闭窗口时优先考虑“隐藏到托盘”而不是直接退出

## 6. 相比上游 Kite 的核心变化

后续 AI 默认应先接受这些事实：

- 当前仓库是一个深度桌面化分叉，不是简单套壳。
- 一些 Web 时代的登录、管理、后台配置能力，在桌面模式下是被裁剪或弱化的。
- 桌面模式下的本地用户身份由后端注入，而不是前端伪造。
- 集群切换、本地路径、原生文件访问、更新器、托盘和应用生命周期，已经变成一等公民。

如果上游 Web 方案与桌面产品方向冲突，默认优先服从桌面版产品方向。
