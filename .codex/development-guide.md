# 开发约定

## 1. 目录导览

- `desktop/`
  - Wails 桌面宿主、bridge、原生集成、更新系统相关接入点。
- `desktop/frontend/`
  - Wails 壳层前端。
- `ui/`
  - 主业务 React 应用。
- `internal/server/`
  - 后端启动、Gin 引擎、路由和运行时装配。
- `pkg/handlers/`
  - HTTP 接口处理器。
- `pkg/handlers/resources/`
  - Kubernetes 资源相关的列表、详情、搜索等处理逻辑。
- `pkg/cluster/`
  - 集群管理与集群生命周期。
- `pkg/model/`
  - 数据模型与持久化。
- `pkg/ai/`
  - AI 对话与工具调用逻辑。
- `docs/`
  - 设计说明、桌面边界、改造计划等文档。
- `cmd/updater/`
  - 桌面端更新器可执行程序。

## 2. 改动前先判断什么

开始实现前，先检查：

1. 这次改动是桌面专属、桌面 / Web 共享，还是遗留 Web 模式逻辑？
2. 逻辑应当落在 `ui/`、后端 API，还是 `desktop/` 宿主层？
3. 这个功能是否需要原生文件 / 路径 / 窗口 / 剪贴板 / 托盘 / 更新器能力？
4. 本地状态在集群切换和应用重启后是否还能正确工作？
5. 如果运行时契约变了，文档是否也要更新？

## 3. 前端约定

- 主业务前端在 `ui/`，不是 `desktop/frontend/`。
- 如果改动的是桌面专属行为，不要只做浏览器层面的临时补丁。
- 涉及桌面能力时，优先走 `ui/src/lib/desktop.ts` 里的封装。
- 运行时分支尽量集中处理，不要散落在很多页面里。
- 如果桌面和 Web 行为不同，默认处理顺序是：
  1. 先在运行时服务或后端运行时层做归一
  2. 再暴露清晰的 capability / runtime contract
  3. 最后让 UI 消费这个契约

## 4. 后端约定

- 不要默认后端只服务远程浏览器访问。
- 桌面启动时会设置 `APP_RUNTIME`、`HOST`、`PORT`、默认 `DB_DSN` 等运行时环境变量。
- 任何影响启动、本地路径、应用生命周期的改动，都应从桌面入口向内检查：
  - `desktop/main.go`
  - `desktop/host.go`
  - `desktop/bridge.go`
  - `internal/server/`

新增或修改 API 时，请额外判断：

- 这是桌面专属接口、共享接口，还是遗留的 Web 语义接口
- 桌面专属能力是否通过显式接口 / 契约暴露
- 如果运行时契约有变化，是否需要同步更新文档

## 5. 持久化与本地状态

- 桌面模式默认把数据存到用户本地配置目录下的 `Kite/`。
- 默认 SQLite 路径是 `Kite/kite.db`，除非显式设置了 `DB_DSN`。
- 收藏、侧边栏偏好这类桌面本地偏好应优先走后端偏好接口并持久化到 SQLite。
- 前端仍会通过 `localStorage` 保存一部分轻量状态或迁移标记，很多时候还会按当前集群做分桶。
- 修改本地状态逻辑时，必须同时考虑：
  - 集群切换后的行为
  - 应用重启后的持久化
  - 内存状态与本地存储之间是否会不一致

## 6. 标准开发命令

默认以仓库根目录作为工作目录。

- 安装依赖：`make deps`
- 启动桌面开发环境：`make dev`
- 构建桌面应用：`make build`
- 构建共享前端静态资源：`make static`
- 运行测试：`make test`
- 运行 lint：`make lint`
- 代码格式化：`make format`

补充说明：

- `make dev` 是默认的桌面开发入口。
- `make build` 是默认的桌面构建入口。
- `make static` 会构建 `ui/` 下的共享业务前端资源，供桌面应用嵌入 / 加载。
- 除非有充分理由，否则不要自行发明新的本地开发主流程。

## 7. 测试要求

完成任务前，应尽量验证对应层级：

- 后端改动：
  - `go test ./...`
- UI 改动：
  - `pnpm --dir ui run test`
- 混合改动：
  - `make test`

如果无法完整验证，需要明确说明哪些没有执行，以及原因是什么。

## 8. 常见误区

- 把 `main.go` 当成唯一重要入口。
- 用纯浏览器 API 实现桌面功能。
- 实际该改 `ui/`，却误改了 `desktop/frontend/`。
- 把上游 Web 项目的登录 / 管理假设重新带回桌面模式。
- 到处分散写运行时判断，而不是走统一契约或能力判断。
- 使用 `localStorage` 时忽略了集群维度。
- 默认认为上游 Kite 的原始实现一定仍然适合当前分叉仓库。

## 9. 文档维护要求

当你发现以下任一内容发生变化时，应同步更新文档：

- 项目运行方式变化
- 桌面能力接入方式变化
- 关键目录职责变化
- 标准开发命令变化
- 运行时契约变化

优先更新：

1. `.codex/` 下文档
2. 对应 `docs/*.md`
3. 必要时更新 `AGENTS.md`
