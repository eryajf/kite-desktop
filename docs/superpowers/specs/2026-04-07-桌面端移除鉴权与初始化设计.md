# 桌面端移除鉴权与初始化设计

## 概述

本文档用于定义如何把当前仓库里遗留的 Web 管理台式鉴权、初始化、以及 Kite 自身权限体系，从桌面端产品中彻底移除。

目标产品模型如下：

- Kite 桌面端是一个本地优先的应用。
- 用户打开应用后直接进入产品。
- 首次启动时可以没有任何集群配置。
- 集群配置通过应用内的“设置”完成。
- 桌面端不再存在登录、超级用户初始化、OAuth、LDAP、API Key、Kite 自身 RBAC 这些产品概念。

本次目标是“彻底移除”，但执行方式采用分阶段推进，确保每个阶段完成后桌面端都仍然可运行、可验证。

## 当前问题

仓库里当前混杂着两套产品模型：

- 旧的服务端部署模型，要求先完成认证、用户初始化、以及集群导入。
- 新的桌面端模型，已经通过本地运行时判断绕过了一部分登录逻辑，但仍然保留了不少旧模型残留。

这些残留体现在：

- 前端仍然存在 `/login`、`/setup` 等历史入口。
- 路由守卫仍然会基于初始化状态做拦截。
- 设置页仍然暴露认证、RBAC、用户、API Key、审计等服务端管理能力。
- 后端仍然保留 `init_check`、`/api/auth/*`、`create_super_user` 等旧初始化与鉴权接口。
- 保留中的资源接口和处理器，很多仍依赖 `RequireAuth`、`RequireAdmin`、`user` 上下文和 Kite 自身 RBAC 校验。

这导致桌面端在产品形态上仍没有真正摆脱旧服务端模式。

## 目标

- 移除“登录”和“初始化”作为桌面端产品概念。
- 让桌面端启动后直接进入主应用。
- 支持“首次启动没有任何集群”这一正常状态，并在应用内明确引导用户去设置中配置集群。
- 从桌面端界面中移除 Kite 自身的认证、用户、OAuth、LDAP、API Key、审计管理、RBAC 管理能力。
- 从桌面端主运行链路中移除后端鉴权、初始化、Kite 自身 RBAC 依赖。
- 保留桌面端真正需要的核心能力：集群管理、资源浏览、日志、终端、监控、搜索、模板、桌面原生能力。

## 非目标

- 不重做 Kubernetes 资源页的信息架构。
- 不移除 Kubernetes 自身的 RBAC 资源浏览能力，例如 `roles`、`rolebindings`、`clusterroles`、`clusterrolebindings` 这些资源对象。
- 不为桌面端引入新的多用户模型。
- 不保留与旧服务端启动流程的兼容层。

## 已确认方向

已确认采用“分阶段完成彻底移除”的方式推进：

1. 先让桌面端主运行链路摆脱 auth/init/RBAC 依赖。
2. 再删除前端旧入口、旧页面、旧设置能力。
3. 再删除后端旧接口、旧中间件、旧处理器注册。
4. 最后清理测试、文档和剩余引用。

这不是“双模式并存”的方案，最终形态是统一的桌面端产品模型，而不是在桌面端外面再套一层旧服务端兼容逻辑。

## 架构设计

### 一、运行时模型

桌面端应明确建模为“本地应用 + 可选集群连接”。

必须成立的运行时假设：

- 应用启动后直接进入 `/`。
- 不存在任何需要登录才能进入的路由。
- 不存在任何需要初始化引导才能进入的路由。
- 正常桌面端请求不依赖 auth cookie、auth token 或 user 对象。
- 桌面端功能不再基于 admin 角色、provider 类型之类用户概念分支。

桌面端唯一合理的前置条件应当是“是否已有集群配置”：

- 如果已有集群，则使用当前集群正常工作。
- 如果没有集群，应用也必须能完整加载，只是在首页展示空状态，并引导用户去设置中添加集群。

### 二、前端边界调整

#### 1. 删除旧路由概念

这些概念不应继续存在于桌面端路由模型中：

- `/login`
- `/setup`
- `ProtectedRoute`
- `InitCheckRoute`

根路由应当直接渲染主应用。

#### 2. 移除鉴权绑定的应用状态

当前 `AuthContext` 混合了两类职责：

- 旧鉴权状态与相关操作
- 桌面端本地运行时判断

这两者需要拆开。最终前端不应再暴露这些与鉴权相关的能力：

- `user`
- `login`
- `logout`
- `refreshToken`
- provider 列表
- 基于 session 检查得出的 loading 状态

桌面端仍然可能需要“当前是否是桌面本地运行时”这一能力，但它应当脱离鉴权上下文，收敛为更小的运行时状态，或者直接按需内联。

#### 3. 把“无集群”变成一等状态

前端需要把 `clusters.length === 0` 视为一个合法且稳定的产品状态，而不是错误态。

无集群时的期望行为：

- 应用壳仍然渲染。
- 侧边栏仍然渲染。
- 集群选择器显示明确的空状态文案。
- 首页不发起依赖集群的概览请求，而是展示空状态卡片。
- 如果用户手动进入依赖集群的页面，应展示“请先配置集群”的提示，而不是登录错误或权限错误。

#### 4. 把设置页收缩为桌面端配置中心

设置页只保留桌面端仍然需要的内容：

- desktop
- general
- clusters
- templates

需要删除的设置能力：

- authentication
- RBAC management
- user management
- API key management
- audit log management

任何基于 `isLocalMode`、`user`、admin 角色进行分支的设置页逻辑，都应改成桌面端固定行为。

### 三、后端边界调整

#### 1. 简化 API 路由结构

`internal/server/routes.go` 不应再按照以下方式组织桌面端 API：

- auth routes
- admin routes
- protected routes

桌面端后端应只暴露本地产品仍然需要的接口。

#### 2. 删除旧初始化与鉴权接口

这些接口族需要移除：

- `/api/v1/init_check`
- `/api/auth/*`
- `/api/v1/admin/users/create_super_user`
- Kite 用户管理接口
- OAuth provider 管理接口
- LDAP 设置接口
- API Key 管理接口
- Kite 自身 RBAC 角色管理接口
- 仅服务于旧用户/权限模型的审计管理接口

#### 3. 从保留接口中移除鉴权中间件依赖

当前保留中的集群与资源接口，大多仍依赖以下机制：

- `RequireAuth()`
- `RequireAdmin()`
- 在请求上下文里注入 `user`
- 在处理器内部使用 Kite 自身 RBAC 做权限判断

这些依赖必须从桌面端保留接口中剥离，包括：

- 路由层认证中间件
- 路由层管理员中间件
- 基于 `user` 或 Kite 角色分配的处理器级权限判断

桌面端保留下来的后端能力，应是“围绕集群工作”的模型，而不是“围绕用户身份工作”的模型。

#### 4. 保留 Kubernetes 资源能力

Kubernetes 资源、日志、终端、监控、搜索等桌面端核心能力仍然保留。

但它们的访问模型从：

- “这个 Kite 登录用户是否有权限访问该资源”

改为：

- “当前是否已选中有效集群，以及这次资源操作本身是否有效”

### 四、数据与模型清理

当主运行链路完成去耦之后，只服务于旧模型的代码都应删除，包括：

- 前端页面
- 前端组件
- 前端 API hooks 与类型
- 后端处理器
- 后端中间件
- 后端 auth 包
- 与用户、OAuth、LDAP、API Key、Kite 自身 RBAC 相关的模型使用点
- 对应测试
- 对应多语言文案
- 对应文档

需要特别强调：

Kubernetes 资源里的 RBAC 对象与 Kite 自身权限系统不是一回事。前者保留，后者删除。

## 分阶段实施

### 阶段一：主运行链路去耦

目标：
先让桌面端在不依赖 auth/init/RBAC 的情况下稳定运行，再进入大规模删除阶段。

范围：

- 去掉桌面端启动流程里的前端路由守卫。
- 去掉前端应用壳对 auth context 的依赖。
- 引入稳定的无集群空状态。
- 去掉桌面端保留 API 上的后端 auth/admin 路由依赖。
- 去掉桌面端保留处理器里的 Kite 自身 RBAC 校验。

完成标准：

- 桌面端启动后直接进入 `/`。
- 不再跳转 `/login` 或 `/setup`。
- 正常桌面端使用不依赖 auth cookie 或 token。
- 无集群时应用内展示空状态并引导去集群设置。

### 阶段二：产品面清理

目标：
在主链路已不再依赖旧模型后，删除前端残留的旧入口、旧页面、旧设置能力。

范围：

- 删除登录页与其测试。
- 删除初始化页与其测试。
- 删除 auth/init 相关路由。
- 删除设置页中的认证、用户、RBAC、API Key、审计管理 tab 与相关弹窗。
- 删除对应的 API hooks、类型定义和多语言文案。
 
完成标准：

- 桌面端界面中不再出现登录、初始化、认证、管理员管理等产品面。
- 设置页只保留桌面端相关内容。
- 前端主运行链路不再依赖 `useAuth`、`ProtectedRoute`、`InitCheckRoute`。

### 阶段三：后端能力删除

目标：
删除那些只服务于旧 Web 管理台模型的后端能力。

范围：

- 删除 auth 路由注册。
- 删除 init 路由注册。
- 删除用户、OAuth、LDAP、API Key、Kite 自身 RBAC 管理路由注册。
- 删除已经无引用的处理器与中间件。
- 移除随之变成死代码的包依赖。

完成标准：

- 后端不再暴露旧鉴权、旧初始化、旧管理端接口。
- 保留下来的 API 编译通过，且运行时不需要 auth 包参与。
- 桌面端主运行路径完全独立于用户身份、session、token。

### 阶段四：清理与验证

目标：
补齐剩余测试与文档，确保新的桌面端产品模型完整闭环。

范围：

- 清理死代码与无用导入。
- 更新文档，明确桌面端已改为直入式产品模型。
- 增补或调整无集群启动、设置入口等关键测试。
- 通过 `make dev` 完成手工验证。

完成标准：

- 仓库中不再残留桌面端无关的登录/初始化产品概念引用。
- 关键桌面端链路具备测试覆盖。
- 手工验证通过。

## 风险与规避

### 风险一：隐式的 user 上下文耦合

不少处理器当前之所以能正常工作，是因为认证中间件提前把 `user` 注入了上下文。一旦直接删除 auth 流程，很可能出现编译通过但运行时报错的问题。

规避方式：

- 先做“保留链路去 user 化”，再删 auth 包和旧处理器。
- 在删除前系统性搜索 `user`、`RequireAuth`、`RequireAdmin`、Kite RBAC 校验调用点。

### 风险二：只删入口，不删内部依赖

如果只去掉 `/login` 和 `/setup`，但首页、设置页、header、hint 等组件还在依赖用户状态，桌面端仍会在运行时表现异常。

规避方式：

- 把应用壳、首页、设置页视为同一批次去耦对象。
- 不允许“路由放开了，但内容层还在读取 user/admin 状态”的半完成状态存在。

### 风险三：Kubernetes RBAC 与 Kite RBAC 混淆

仓库里“rbac”一词同时指向 Kubernetes 资源对象和 Kite 自身权限体系。如果清理边界不清，很容易误删 Kubernetes 资源能力。

规避方式：

- 只删除 Kite 自身权限系统。
- 保留 Kubernetes RBAC 资源对象的浏览与操作能力。

### 风险四：改动面大，回归面广

这次改动会同时影响：

- 前端路由
- 前端上下文
- 前端设置页
- 前端 API hooks
- 后端路由
- 后端中间件
- 后端处理器
- 测试
- 文档

规避方式：

- 按阶段提交，不做一次性大爆破。
- 每一阶段结束后都验证桌面端能否正常启动和进入关键路径。

## 测试策略

测试不能等到全部删除完成后再统一补，必须跟随阶段推进。

关键验证点：

- 前端路由：桌面端启动后不再被登录或初始化逻辑重定向。
- 前端无集群状态：首页在无集群时展示空状态，而不是发起错误请求。
- 设置页：只保留桌面端真正需要的 tab。
- 后端保留接口：桌面端关键接口在无 auth/session 前提下可正常访问。
- 手工链路：通过 `make dev` 验证启动、进入首页、进入设置、添加集群等核心路径。

建议补充的定向测试：

- 根路由直入测试。
- 首页无集群空状态测试。
- 集群选择器空状态测试。
- 设置页 tab 收缩测试。
- 后端保留 API 无鉴权依赖测试。

## 影响范围概览

前端主要影响范围：

- 路由定义
- 应用 providers 与应用壳
- 运行时上下文
- 首页
- 集群选择器
- 设置页及其子组件
- 与 init/auth/admin 相关的 API hooks
- 对应多语言文案与测试

后端主要影响范围：

- 服务端路由注册
- auth/admin 中间件引用
- 含 Kite RBAC 校验的处理器
- init/auth/user/OAuth/LDAP/API Key/Kite RBAC 相关处理器注册
- 对应测试与辅助包

## 验收标准

最终实现完成后，应满足以下条件：

- 打开桌面端应用时不再需要登录或初始化。
- 用户首次看到的空状态是“尚未配置集群”，而不是“创建用户”或“登录”。
- 用户通过设置页完成集群配置。
- 界面中不再存在旧鉴权/旧管理台相关产品面。
- 桌面端后端运行路径不再依赖用户身份、token、Kite 自身角色。
- 已配置集群时，Kubernetes 资源管理相关能力仍可正常工作。

本文档明确采用“分阶段推进，但目标彻底移除”的执行策略。这就是本次设计的最终结论。
